## 产品定义

这是一个“恋爱可能性实验室”，不是单纯的配对打分器。

用户一方面扮演恋综中的主角嘉宾，带入自己进入节目；另一方面又是实验者，可以在不污染主线人格的前提下，创建多个实验分支，测试“如果我更主动一点”“如果我更坦诚一点”“如果我少一点防御”会发生什么。

产品的核心价值不是给出一个静态的恋爱概率，而是让用户看到：

1. 关系是怎么一步步变化的
2. 哪些行为和策略推动了关系升温或降温
3. 为什么这次没有谈上
4. 下次应该尝试什么不同的做法

## 用户角色

主用户是希望“代入自己做恋爱实验”的人，而不是纯旁观分析师。

用户在产品里同时拥有两种身份：

1. 主角嘉宾：把自己投放进恋综环境，观察自己和不同嘉宾的关系走向
2. 实验者：在实验分支中修改主角人格或行为策略，探索不同可能性

## 世界规则

### 1. 主线和实验分支分离

采用双轨制：

1. 主线人格：长期保存，代表用户当前正式设定
2. 实验分支：每次实验复制一份主线人格，在分支中修改人格和策略，模拟结束后不写回主线

这样用户既能沉浸，又能反复试错，不会把结果玩乱。

### 2. 用户只做中度干预

用户不逐句操控对话，也不当提线木偶。

每个场景开始前，用户只选择一张或多张“策略卡”，例如：

1. 更主动表达兴趣
2. 少试探，多直接确认
3. 先安抚情绪，再推进关系
4. 展示价值观，不急于暧昧
5. 避免争夺注意力

之后由 Agent 在既定人格、场景目标和当前关系状态下自主演化。

## 输入系统

### 1. 支持输入

1. 微信聊天记录
2. 嘉宾背景资料
3. 用户补充描述文档

### 2. 嘉宾背景必须包含的信息

不能只有人口学信息，至少要包含：

1. 基础档案：年龄、职业、城市、教育、外形标签
2. 情感史：恋爱经历、创伤、依恋模式、信任建立方式
3. 择偶标准：外形偏好、价值观偏好、互动偏好、雷点
4. 性格标签：外向/内向、主动/被动、冲突回避/正面应对
5. 节目目标：想认真恋爱、想先观察、想被看见、想追特定类型

### 3. 输入后的系统动作

1. 提取嘉宾与关系对象
2. 为每位嘉宾构建独立 `Soul.md`
3. 生成初始关系图
4. 基于择偶标准和背景资料计算初始吸引力与初始风险点

## Soul.md 设计

`Soul.md` 不是人物简介，而是“可被模拟更新的人物状态文件”。

每个 Agent 至少包含两类字段：

### 1. 稳定层

这些字段默认不随单轮对话快速变化：

1. 核心性格
2. 依恋风格
3. 择偶标准
4. 雷点与边界
5. 表达偏好
6. 冲突处理方式

### 2. 可变层

这些字段会在场景中持续更新：

1. 对每位对象的亲密度
2. 对每位对象的信任度
3. 对每位对象的吸引力
4. 冲突累积值
5. 焦虑值/不安全感
6. 被理解感
7. 期待值与失望值
8. 当前关系判断：观察中、升温中、摇摆、降温、基本出局

## 单局玩法循环

一次完整模拟不是“10-20 轮抽象聊天”，而是一季恋综缩略版。

单局固定为一组有戏剧张力的节目场景：

1. 破冰初见
2. 第一次自由交流
3. 双人约会选择
4. 多人晚餐与暗流
5. 私聊澄清或深夜谈心
6. 夜间短信/匿名心动表达
7. 冲突事件或误会事件
8. 关键选择夜
9. 最终告白/关系结算

每个场景都包含 4 个元素：

1. 场景目标：这个场景主要推动什么关系问题
2. 参与对象：谁会出现在这个场景中
3. 用户策略卡：用户在场景开始前的有限干预
4. 状态更新：场景结束后更新 `Soul.md` 与关系图

## 反馈系统

每个场景结束后，不展示完整底层数值，而是给“半透明反馈”。

用户能看到：

1. 当前关系走向卡片：升温、停滞、误会、拉近、失衡
2. 1-2 条关键原因：例如“你更主动后，对方的防御下降”“你在多人场景中过度表现，导致目标对象产生不安全感”
3. 关键事件回顾：本场景发生了什么，为什么重要

用户看不到：

1. 全量底层变量
2. 所有 Agent 的完整内心账本

这样既保留恋综叙事感，也不会让系统变成 Excel。

## 结果页设计

最终结算不做成简单排行榜，而做成“关系故事结算”。

结果页至少输出：

1. 最终关系走向
2. 3 个关键转折点
3. 为什么没谈上，或为什么谈上了
4. 主角最有效的 1-2 个策略
5. 下次值得尝试的变化方向

结果页可以附带一个简化概率层，但概率只能做辅助，不是主角。

## 核心乐趣

这个产品的乐趣不在“算命”，而在“比较不同版本的自己会带来怎样的关系命运”。

因此复玩循环应该是：

1. 建立主线人格
2. 跑出一版默认世界线
3. 看见失败原因或成功原因
4. 创建实验分支，修改人格或策略
5. 重跑关键场景
6. 对比不同分支的关系变化
7. 得到“原来问题不只是对方不喜欢我，而是我的互动方式一直在制造距离”的洞察

## 恋爱可能性的定义

“恋爱可能性”不能只靠一句 Prompt 主观判断，应该拆成两层：

### 1. 结构化关系判断

由以下维度共同决定：

1. 初始吸引力匹配
2. 价值观兼容度
3. 互动舒适度
4. 信任增长速度
5. 冲突后的修复能力
6. 竞争关系压力
7. 关系推进意愿是否对齐

### 2. 叙事化解释

系统用自然语言解释为什么这个结果成立，例如：

1. 对方一开始不在你的择偶范围内，但因为你持续提供安全感，信任快速升高
2. 你们互有吸引，但推进节奏错位，导致在关键节点错过
3. 你在多人竞争场景中过于焦虑，削弱了原本已经建立的好感

## 模式定义

### 1. 主线模式

1. 使用正式主线人格
2. 只记录正式世界线
3. 用于沉浸式体验和长期观察

### 2. 实验模式

1. 复制主线人格生成分支
2. 允许修改人格参数或策略偏好
3. 记录本次实验的关键差异
4. 支持和主线结果进行对比

## 报告输出

报告不应只是“匹配度报告”，而应是“关系诊断报告”。

建议输出：

1. 每位目标嘉宾的最终关系状态
2. 每段关系的关键上升/下降节点
3. 用户最常见的自我阻碍模式
4. 最适合当前人格的恋爱推进方式
5. 如果目标是提高成功率，下一轮最值得改变的一个变量

## 当前版本逻辑已闭环的部分

以下闭环已经基本成立：

1. 输入资料
2. 构建 Agent 与 `Soul.md`
3. 放入恋综场景中演化
4. 场景后更新关系状态
5. 输出关系诊断与实验建议
6. 用户基于建议创建新分支再实验

## 当前版本仍不完整的关键缺口

当前方案还不是 100% 完整，但“产品规格层”已经基本齐了，剩下的是“实现层”定义：

1. 如何从微信聊天记录和背景文档中稳定抽取结构化字段
2. 多 Agent 对话生成时如何保证每个 Agent 不会说崩人格
3. 场景中的事件标签如何从对话中自动归纳
4. 规则分和 LLM 自由判断如何混合，避免结果过死或过飘
5. 前端结果页如何展示分支对比，既清晰又不变成表格工具
6. 如何设计日志与回放，保证每次结算都能追溯原因
