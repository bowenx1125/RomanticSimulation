## 场景设计表

这份文档定义一局恋综模拟的场景结构。

目标不是写剧情脚本，而是给系统一个可以重复运行的“节目骨架”。每个场景都要能回答 5 件事：

1. 为什么会发生这个场景
2. 谁会参与
3. 用户能做什么有限干预
4. 场景会改动哪些关系变量
5. 用户在场景结束后会看到什么反馈

## 单局总结构

建议一局固定为 9 个场景：

1. `scene_01_intro` 破冰初见
2. `scene_02_free_talk` 第一次自由交流
3. `scene_03_date_pick` 双人约会选择
4. `scene_04_group_dinner` 多人晚餐与暗流
5. `scene_05_private_talk` 私聊澄清或深夜谈心
6. `scene_06_signal_night` 夜间短信/匿名心动表达
7. `scene_07_conflict_event` 冲突或误会事件
8. `scene_08_decision_night` 关键选择夜
9. `scene_09_final_confession` 最终告白与关系结算

## 全局规则

### 1. 用户干预规则

每个场景开始前，用户最多选择：

1. 1 张主策略卡
2. 1 张辅助策略卡

用户不能直接写台词，不能直接指定结果。

### 2. 输出规则

每个场景结束后，系统必须输出：

1. 场景摘要
2. 关系走向卡片
3. 1-2 条关键原因
4. 下个场景的 tension 提示

### 3. 跳出规则

即使某段关系接近出局，也不应过早从节目中完全删除，除非：

1. `trust <= 15` 且 `conflict >= 75`
2. 连续 3 个关键场景都没有任何正向事件
3. 一方明确进入 `out` 状态且对另一方没有残余好奇心

## 场景明细

### Scene 01: 破冰初见

| 项目 | 设计 |
|------|------|
| 场景 ID | `scene_01_intro` |
| 目标 | 建立第一印象和初始吸引力 |
| 核心 tension | 外形偏好和第一反应是否匹配 |
| 输入 | 嘉宾背景、择偶标准、初始情绪状态 |
| 参与者 | 全体嘉宾 |
| 用户可选策略卡 | `warm_presence`、`playful_opening`、`quiet_observation` |
| 主要更新变量 | `initial_attraction`、`curiosity`、`comfort` |
| 结束条件 | 所有主要对象完成首次互动或观察 |
| 用户反馈 | “谁对你有第一眼兴趣”“谁只把你当普通人看” |

设计要求：

1. 不要让这个场景决定最终关系，只决定初始偏置
2. 要允许“第一印象一般，但后续逆转”
3. 如果用户策略卡和人格严重冲突，需要在解释中体现违和感

### Scene 02: 第一次自由交流

| 项目 | 设计 |
|------|------|
| 场景 ID | `scene_02_free_talk` |
| 目标 | 从外形吸引转向互动舒适度 |
| 核心 tension | 谁聊得来，谁只是表面礼貌 |
| 输入 | Scene 01 结果、用户策略卡 |
| 参与者 | 主角 + 2-3 位潜在对象 |
| 用户可选策略卡 | `seek_common_ground`、`ask_deeper_questions`、`stay_mysterious` |
| 主要更新变量 | `comfort`、`understood`、`curiosity` |
| 结束条件 | 每个目标对象至少完成一段有效交流 |
| 用户反馈 | “谁和你聊得顺”“谁开始误解你” |

设计要求：

1. 这个场景是筛出潜在主线关系的第一步
2. 允许“聊得开心但并不来电”
3. 被理解感要和舒适度分开计算

### Scene 03: 双人约会选择

| 项目 | 设计 |
|------|------|
| 场景 ID | `scene_03_date_pick` |
| 目标 | 放大偏好与被选择感 |
| 核心 tension | 你想选的人会不会也选你 |
| 输入 | 前两场的关系趋势、节目规则 |
| 参与者 | 主角 + 所有可约会对象 |
| 用户可选策略卡 | `be_direct`、`wait_for_signal`、`choose_safe_option` |
| 主要更新变量 | `expectation`、`trust`、`anxiety`、`exclusivity_pressure` |
| 结束条件 | 约会对象配对完成 |
| 用户反馈 | “互选、单选落空、保守错过”的结果解释 |

设计要求：

1. 互选要明显加分
2. 落选不应只掉吸引力，也应提高焦虑或失望
3. 这个场景是第一个强转折点

### Scene 04: 多人晚餐与暗流

| 项目 | 设计 |
|------|------|
| 场景 ID | `scene_04_group_dinner` |
| 目标 | 暴露竞争关系和社交表现 |
| 核心 tension | 多人场景下的安全感和嫉妒 |
| 输入 | 当前潜在关系、竞争图谱 |
| 参与者 | 全体嘉宾 |
| 用户可选策略卡 | `hold_center`、`focus_one_person`、`avoid_competition` |
| 主要更新变量 | `anxiety`、`conflict`、`exclusivity_pressure`、`trust` |
| 结束条件 | 晚餐互动结束并形成新的印象偏置 |
| 用户反馈 | “你在竞争中显得更有魅力，还是更不稳定” |

设计要求：

1. 这是最容易出误会的场景
2. 多人场景中的策略不应该完全复制私聊规则
3. 同时要给非目标对象留下观察视角

### Scene 05: 私聊澄清或深夜谈心

| 项目 | 设计 |
|------|------|
| 场景 ID | `scene_05_private_talk` |
| 目标 | 修复误会或建立深层连接 |
| 核心 tension | 敢不敢把真实需求说出来 |
| 输入 | 晚餐后的 tension、场景分配 |
| 参与者 | 主角 + 1 位关键对象 |
| 用户可选策略卡 | `reassure_first`、`state_needs_clearly`、`listen_without_defending` |
| 主要更新变量 | `trust`、`understood`、`anxiety`、`conflict` |
| 结束条件 | 私聊结束并完成一次情感交换 |
| 用户反馈 | “关系被修复了，还是只是表面缓和” |

设计要求：

1. 这是信任增长最强的场景之一
2. 如果用户继续回避核心问题，系统要明确指出
3. 修复不等于立刻升温，可能只是止损

### Scene 06: 夜间短信/匿名心动表达

| 项目 | 设计 |
|------|------|
| 场景 ID | `scene_06_signal_night` |
| 目标 | 让隐性偏好变成显性信号 |
| 核心 tension | 你以为的双向，是否真的双向 |
| 输入 | 当前关系热度、当晚事件情绪 |
| 参与者 | 全体嘉宾 |
| 用户可选策略卡 | `send_clear_signal`、`keep_ambiguity`、`protect_self_image` |
| 主要更新变量 | `expectation`、`disappointment`、`trust` |
| 结束条件 | 信号发送并完成结果揭示 |
| 用户反馈 | “谁把你放在心上”“谁让你误判了关系” |

设计要求：

1. 这个场景专门制造高期待和高落空
2. 夜间信号不应单独决定结局，但应改变后续推进意愿

### Scene 07: 冲突或误会事件

| 项目 | 设计 |
|------|------|
| 场景 ID | `scene_07_conflict_event` |
| 目标 | 检验关系抗压能力 |
| 核心 tension | 关系在摩擦面前是升级还是塌陷 |
| 输入 | 既有 tension、节目事件卡、竞争图谱 |
| 参与者 | 主角 + 1-2 位关键对象 |
| 用户可选策略卡 | `repair_quickly`、`stand_ground_calmly`、`retreat_and_observe` |
| 主要更新变量 | `conflict`、`trust`、`anxiety`、`comfort` |
| 结束条件 | 冲突事件落地并形成修复或裂痕 |
| 用户反馈 | “你是修复了伤害，还是让对方更失望” |

设计要求：

1. 冲突必须能解释成具体原因，不能凭空掉分
2. 一个关系能否修复，应该成为最终结算的重要依据

### Scene 08: 关键选择夜

| 项目 | 设计 |
|------|------|
| 场景 ID | `scene_08_decision_night` |
| 目标 | 收敛摇摆，逼近最终选择 |
| 核心 tension | 你会继续赌一个不确定对象，还是转向更稳的人 |
| 输入 | 所有关系的当前状态和趋势 |
| 参与者 | 主角 + 主要候选对象 |
| 用户可选策略卡 | `commit_to_one`、`seek_final_clarity`、`reduce_risk` |
| 主要更新变量 | `commitment_alignment`、`expectation`、`anxiety` |
| 结束条件 | 主角和主要对象做出明确选择倾向 |
| 用户反馈 | “你在这一夜是变得更坚定，还是更摇摆” |

设计要求：

1. 必须让用户感到有代价
2. 保守路线和冒险路线都要有成立可能

### Scene 09: 最终告白与关系结算

| 项目 | 设计 |
|------|------|
| 场景 ID | `scene_09_final_confession` |
| 目标 | 产出可解释的关系结局 |
| 核心 tension | 情感是否足够稳定到进入关系 |
| 输入 | 全局关系变量、关键记忆、趋势线 |
| 参与者 | 主角 + 最终对象或多个未收敛对象 |
| 用户可选策略卡 | 无新增策略卡，只结算 |
| 主要更新变量 | `status`、最终关系归因 |
| 结束条件 | 输出最终关系故事结算 |
| 用户反馈 | 最终走向、3 个关键转折点、失败/成功原因、下次建议 |

设计要求：

1. 结算必须回溯到前面具体场景
2. 不允许只输出“匹配度 78 分”
3. “没谈上”也必须有高价值解释

## 策略卡可用性表

| 策略卡 | 适用场景 | 作用方向 |
|--------|----------|----------|
| `warm_presence` | 01 | 降低陌生感，提高舒适度 |
| `playful_opening` | 01 | 提高好奇心和初始吸引 |
| `quiet_observation` | 01 | 降低失误，但也可能错过存在感 |
| `seek_common_ground` | 02 | 提高舒适度和被理解感 |
| `ask_deeper_questions` | 02,05 | 提高被理解感和信任 |
| `stay_mysterious` | 02 | 提高好奇心，但可能压低安全感 |
| `be_direct` | 03 | 提高明确性，降低模糊成本 |
| `wait_for_signal` | 03 | 降低风险，但可能错过互选 |
| `choose_safe_option` | 03 | 选择稳定对象而非心动对象 |
| `hold_center` | 04 | 保持社交稳定，减少失态 |
| `focus_one_person` | 04 | 向目标对象传递明确偏好 |
| `avoid_competition` | 04 | 降低焦虑，但也可能失去存在感 |

| `state_needs_clearly` | 05,08 | 提高明确性和关系对齐度 |
| `listen_without_defending` | 05,07 | 降低冲突，增加被理解感 |
| `send_clear_signal` | 06 | 提高期待，也提高落空风险 |
| `keep_ambiguity` | 06 | 保留退路，但会降低推进效率 |

| `reduce_risk` | 08 | 减少失败感，但可能降低戏剧性 |

## 分支与复玩规则

1. 一局结束后，用户可以基于任一关键场景创建实验分支
2. 创建分支时，默认保留之前场景结果，只重跑选中的关键场景及其后续场景
3. 如果改的是稳定层人格，建议整局重跑，不允许只重跑后半段
4. 如果改的是策略卡，只重跑对应场景及其后续场景

## 前端最小展示结构

每个场景建议前端至少有 4 个区域：

1. 场景标题和 tension 简介
2. 参与角色和当前关系提示
3. 用户策略卡选择区
4. 场景结果反馈区

这样产品体验会更像“看自己出演一季恋综”，而不是“浏览一堆模拟日志”。
